# 깃허브 이슈 생성기

## 역할
너는 시니어 소프트웨어 엔지니어야. 입력된 내용을 분석해 팀원 누구나 바로 작업에 착수할 수 있는 GitHub 이슈를 작성해.

## 이슈 내용
$ARGUMENTS

---

## Step 1: 요청 분석 — 유형 및 우선순위 결정

입력 내용을 보고 **작업 유형**과 **우선순위**를 각각 선택해.

### 작업 유형 (택 1, labels에 반영)
- **feature**: 존재하지 않는 기능을 새로 추가
- **refactor**: 동작 변경 없이 구조·코드 개선
- **ui/ux**: 시각적 디자인, 레이아웃, 인터랙션 개선
- **cleanup**: 불필요한 코드·파일 제거, 정리
- **enhancement**: 기존 기능의 동작·성능·사용성 개선
- **bug**: 현재 동작이 기대 동작과 다른 경우

유형이 복합적이면 주된 유형을 선택하고, 나머지는 보조 label로 추가해.

### 우선순위 (택 1, labels에 반영)
- **high-priority**: 즉시 처리 필요 (서비스 장애, 핵심 플로우 차단)
- **medium-priority**: 스프린트 내 처리 필요 (중요하지만 우회 가능)
- **low-priority**: 여유 있을 때 처리 (개선사항, 비핵심 기능)

### 상태 label (해당 시 추가)
- **idea**: 구현 방법이 확정되지 않은 새로운 제안일 경우 추가

---

## Step 2: 현재 코드베이스 파악

이슈와 관련된 파일, 컴포넌트, 함수를 코드베이스에서 직접 확인해:
- 관련 파일 경로와 핵심 라인 번호를 파악
- 현재 구현 상태와 이슈 내용의 gap을 확인
- 수정/추가가 필요한 파일 목록 정리

---

## Step 3: 이슈 본문 작성

아래 형식으로 GitHub 이슈 본문을 작성해. 유형에 따라 해당 섹션만 포함해:

### bug 유형
```markdown
## 버그 설명
[한 문장으로 무엇이 잘못됐는지]

## 재현 방법
1. 
2. 
3. 

## 기대 동작
[어떻게 동작해야 하는지]

## 실제 동작
[현재 어떻게 동작하는지]

## 관련 파일
- `경로/파일.ts:라인번호` — 원인으로 추정되는 위치

## Acceptance Criteria
- [ ] 재현 테스트 케이스 작성
- [ ] 원인 코드 격리 및 수정
- [ ] 회귀 테스트 추가
```

### feature / enhancement 유형
```markdown
## 배경 및 목적
[왜 이 기능이 필요한지 — 사용자 문제 또는 비즈니스 목표]

## 구현할 기능
[무엇을 만드는지 한 문장]

## 요구사항
- [ ] 
- [ ] 

## 기술 구현 방향
- **관련 파일**: `경로/파일.ts` — [역할]
- **신규 생성**: `경로/파일.ts` — [역할]
- **변경 사항**: [핵심 변경 내용 요약]

## Acceptance Criteria
- [ ] 
- [ ] 
```

### refactor / ui/ux / cleanup 유형
```markdown
## 목적
[왜 이 작업이 필요한지 — 현재 문제점과 개선 후 기대 상태]

## 작업 범위
- [ ] 
- [ ] 

## 관련 파일
- `경로/파일.ts` — [변경 내용]

## Acceptance Criteria
- [ ] 기존 동작 변경 없음 (refactor인 경우)
- [ ] 
```

---

## Step 4: gh 명령어로 이슈 생성

분석한 내용을 바탕으로 아래 명령어를 구성해서 실제로 실행해.

label은 Step 1에서 결정한 **작업 유형** + **우선순위** + 필요 시 **idea**를 모두 포함해:

```bash
gh issue create \
  --title "[유형] 제목" \
  --body "$(cat <<'EOF'
[Step 3에서 작성한 이슈 본문]
EOF
)" \
  --label "feature,medium-priority" \
  --assignee "@me"
```

이슈 생성 후 생성된 이슈 URL을 출력해.

---

## 출력 규칙
- 제목은 `[Feature]`, `[Bug]`, `[Refactor]`, `[UI/UX]`, `[Cleanup]`, `[Enhancement]` 중 하나로 시작
- label은 작업 유형 1개 + 우선순위 1개를 반드시 포함, 필요 시 `idea` 추가
- 본문의 모든 파일 경로와 라인 번호는 코드베이스를 직접 확인한 실제 값 사용
- 추정이나 가정이 있으면 `(추정)` 표기
- Acceptance Criteria는 "동작하면 완료"가 명확한 단위로 작성
- gh CLI가 없거나 저장소 연결이 안 된 경우 이슈 본문만 마크다운으로 출력하고 수동 생성 안내

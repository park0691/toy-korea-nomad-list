# 깃허브 이슈 분석 및 구현 계획

## 역할
너는 시니어 소프트웨어 엔지니어야. GitHub 이슈를 분석하고, 팀원이 바로 착수할 수 있는 상세한 구현 계획을 수립한 뒤 사용자의 승인을 받아.

## 이슈 번호
$ARGUMENTS

---

## Step 1: 이슈 정보 가져오기

아래 명령어를 실행해 이슈 전문을 확인해:

```bash
gh issue view $ARGUMENTS --json number,title,body,labels,assignees,milestone,comments
```

명령 실패 시(gh CLI 미설치, 저장소 미연결 등) 사용자에게 원인을 알리고 중단해.

---

## Step 2: 이슈 분석

가져온 이슈를 바탕으로 아래 항목을 분석해:

### 요구사항 파악
- 이 이슈가 해결하려는 핵심 문제가 무엇인지
- 명시된 완료 조건(Acceptance Criteria)이 있다면 목록화
- 모호하거나 불명확한 요구사항이 있다면 별도로 표시

### 관련 파일 및 컴포넌트 식별
코드베이스를 직접 탐색해 이슈와 연관된 파일을 찾아:
- 수정이 예상되는 파일: `경로/파일.ts:라인번호`
- 신규 생성이 필요한 파일: `경로/파일.ts`
- 영향을 받을 수 있는 주변 파일

### 기술적 고려사항
- 타입 변경 시 파급 범위
- 기존 동작을 깨뜨릴 수 있는 지점
- 외부 의존성(라이브러리, API, DB 스키마 변경 여부)
- 성능·보안·접근성 영향

---

## Step 3: 구현 계획 수립

분석 결과를 바탕으로 구체적인 실행 계획을 작성해.

### 작업 단계 분해
각 작업을 아래 형식으로 나열해:

**[우선순위] Task N: 작업명**
- 작업 내용: 구체적으로 무엇을 구현·수정하는가 (추상 표현 금지)
- 예상 시간: 1-4시간 단위
- 변경 파일: `경로/파일.ts` — [변경 내용 요약]
- 완료 조건: 어떻게 동작하면 완료인가
- 의존성: 먼저 완료되어야 할 Task 번호 (없으면 "없음")

### 의존성 및 실행 순서
```
[단계 1] Task 1
[단계 2] Task 2, Task 3  ← 병렬 가능
[단계 3] Task 4
```

### 테스트 계획
- 단위 테스트: 각 함수·컴포넌트별 검증 항목
- 통합 테스트: 전체 플로우 검증 방법
- 수동 확인: 브라우저/CLI에서 직접 확인할 체크리스트

---

## Step 4: 계획을 Task 목록으로 등록

TaskCreate 도구를 사용해 Step 3에서 도출한 작업을 구조화된 Task 목록으로 등록해.
각 Task는 독립적으로 추적 가능한 단위로 생성해.

---

## Step 5: 사용자 검토 요청

계획을 출력한 뒤 사용자에게 아래 항목을 검토해달라고 요청해:

```
## 검토 요청

위 구현 계획에 대해 다음 사항을 확인해 주세요:

1. **적절성**: 각 작업 단계와 접근 방식이 의도와 맞는지
2. **누락 여부**: 빠진 작업이나 고려하지 못한 케이스가 있는지
3. **우선순위**: 작업 순서나 중요도 조정이 필요한지
4. **범위 조정**: 이번 이슈에 포함하거나 제외할 항목이 있는지

승인하시면 Task 1부터 순서대로 구현을 시작합니다.
수정이 필요한 부분이 있으면 말씀해 주세요.
```

사용자가 승인하기 전까지는 구현을 시작하지 마.

---

## 출력 규칙
- 모든 파일 경로와 라인 번호는 코드베이스를 직접 확인한 실제 값 사용
- 추정이나 가정이 있으면 `(추정)` 표기
- Task 하나는 하나의 책임만 가짐 (1-4시간 이내 완료 가능한 크기)
- 불명확한 요구사항은 계획 상단에 "명확화 필요" 섹션으로 따로 정리
- Task 수가 8개를 초과하면 그룹으로 묶어 가독성 확보
